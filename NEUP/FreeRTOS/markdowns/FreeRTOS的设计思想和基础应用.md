## 1. 背景：为什么我们需要 RTOS？

### 1.1 先说说单片机是怎么跑程序的

在上次的[嵌入式,让我看看!](../../Build_Process/markdowns/嵌入式，让我看看！.md)中 , 静态编译型语言，最后都会通过编译器一步步的编译过程, 把它变成二进制机器码，烧进 Flash，上电后 CPU 一条一条取指令执行。

在裸机单片机上, 程序一般包括以下几个部分:(这里以 经典的按钮点灯 为例)

```c
#include <....>
#include <....>

int main(void) {
    LED_initial();
    kEY_initial();
    SETIAL_initial();//设备初始化, 包括对 部分

    while(1) {
        step1();   // 按键扫描
        step2();   // LED 刷新
        step3();   // 串口发送
    }
}
```

`main()` 进去以后，经过上机紧接着的一系列初始化后(包括startup.s和上文`main`函数内的初始化), 
就是一个`while(1){}` **死循环**，随后CPU的指令读取就在里面一圈一圈地转。这称为 **超级循环（Super Loop）**，是**裸机编程**最典型的样子。

大多数单片机入门都是从这种写法开始的，那这种工程有什么问题吗,
~~当然有, 不然下面讲什么~~
我们接下来分析分析

### 1.2 裸机有啥毛病？

#### 一、CPU 在「等」的时候啥也不干
对上文代码中的循环部分具体化(伪代码如下)
```c
while(1) {
    scan_key();    // 几微秒就扫完了
    delay_ms(10);  // ← CPU 在这 10ms 里啥也没干，纯数数
    toggle_led();
    uart_send();
}
```
> 这里要声明的, 为什么这里需要delay
因为需要`按键去抖`, 
简单说就是, 芯片内部通过读取电平特定跳变, 来判断按键是否被按下, 机械按键按下去的瞬间, 金属触点会像弹跳球一样"弹跳"几次——电平在稳定前会快速通断好几次（一般持续 5~10ms）。如果不 delay 一下等它稳定, CPU 可能把一次按下读成"按了→松了→按了→松了...", 那程序就会认为按钮被按了好几下, 然后LED也开关了好几次

所以用 delay 等 10ms 是合理的——**但问题在于**, CPU 在这 10ms 里**只能空转**, 啥正事也干不了。

为了解决这个问题, 我们要先搞懂——
`delay_ms(10)` 的本质是什么？

```c
void delay_ms(uint32_t ms) {
    for(uint32_t i = 0; i < ms * 1000; i++);
}
```

CPU 在这 10ms 里不停地 `i++`，**别的啥也不干**。按键按了？不知道。传感器来数据了？不知道。中断来了？虽然能响应，但处理完一回来，它继续数数

> 🍳 **类比一下：** 就像厨师在等水烧开的时候，盯着水壶发呆，而不是趁这个时间切菜、洗锅。

#### 二、实时性差 —— 紧急事件得排队等

假设这么个需求：
- 按键按下，必须在 **5ms 内** 响应
- 但 while 循环里有一步特别慢（比如刷 OLED，要 20ms）

```c
while(1) {
    oled_display();    // 20ms , 这时按键按下
    scan_key();        // chovy, 大哥你快点啊， 我花都要谢了（bushi）
    process_sensor();
}
```

<span style="color:red">注：</span> **响应时间不由「紧急程度」决定，由「最长的那个任务」决定。**

> 🚶 **类比一下：** 就像排队办事，前面那个人办了一个超复杂的业务（20分钟），你只是交个表（5秒），但你得等他办完。

#### 三、代码耦合 —— 牵一发而动全身

```c
while(1) {
    scan_key();
    update_led();
    uart_send();
    sensor_read();
    // 想加功能？在这里加一行
    // 想删功能？在这里删一行
    // 所有东西粘在一起！！！
}
```

- 所有功能耦合在同一个 while 里
- 一个模块崩了 → while 卡住 → 整个系统死掉
- 新人看代码：一个 while 里几十个函数调用，想死 😇

#### 四、功耗控制难

做电池设备的时候就哭了。想让 CPU 闲下来省电吧，又怕睡过去醒不来。结果芯片明明没活干，也只能一直跑着。

#### 总结——痛在哪？

| 痛点 | 一句话 |
|------|--------|
| CPU 浪费 | delay 在空转，没干正事 |
| 实时性差 | 紧急事件得等前面的跑完 |
| 代码耦合 | 所有功能塞在一个 while 里 |
| 功耗难降 | CPU 不能闲下来 |
| 难扩展 | 加个功能牵一发动全身 |

那有没有什么东西能解决这些问题呢？当然有 —— 这就是 RTOS 登场的理由。

## 2. FreeRTOS 是什么

### 2.1 一句话 + 详细表格

> **FreeRTOS 是一个为单片机设计的、轻量级的、开源的实时操作系统内核。**
> 这也是我选择讲`freertos`的原因, 它是一种操作系统, 我曾经就想着裸机嵌入式和大家未来方向似乎没有很大关系, 但是`freertos`这种带有操作系统思想的工具, 或许对大家的帮助更大.

| 方面 | 内容 |
|------|------|
| **出身** | 2003 年由 Richard Barry 开发，后来被 Amazon 收购 |
| **名字含义** | **Free**（~~福瑞bushi~~,免费）+ **RTOS**（实时操作系统） |
| **语言** | 纯 C，移植性好 |
| **体量** | 内核最小只需要 **3~5 KB ROM**，RAM 几 KB 就能跑 |
| **芯片支持** | ARM Cortex-M、RISC-V、AVR、PIC、MSP430…… 几乎所有主流单片机 |
| **开源协议** | MIT，商用也没问题 |
| **生态** | 官方有 AWS IoT 集成，社区资源极其丰富，中文资料也多 |

以最经典的 STM32F103C8T6 为例：64KB Flash，20KB SRAM。FreeRTOS 内核占 3~5KB ROM，几百字节 RAM —— **剩下的空间还够写一大堆业务逻辑**。

### 2.2 市面上 RTOS 那么多，为啥选它？

1. **最轻量** —— 占资源是所有 RTOS 里最小的之一
2. **最普及** —— 资料最多，遇到问题一搜就有答案 ~~（虽然可能搜到的是 2015 年的老帖子）~~
3. **大厂背书** —— Amazon 维护，还在持续更新（TrustZone、SMP 多核支持）
4. **完全免费** —— 不像 uC/OS 商用要收费

> 说个不恰当的类比：FreeRTOS 在单片机 RTOS 里的地位，差不多就像 Linux 在服务器操作系统的地位。它不一定是最强最全的，但它是你**大概率会遇到的第一个**。

## 3. 核心设计思想（重点）

好，这部分是今天的主菜，我们一层一层往下剥 🧅

### Q: 等等，上面不是说裸机有那么多毛病吗？那 RTOS 是怎么治这些毛病的？

### A:

| 痛点 | RTOS 怎么解决 |
|------|---------------|
| CPU 浪费 | ✅ Blocked 状态主动交出 CPU（见 3.4） |
| 实时性差 | ✅ 抢占式调度，高优先级任务随时打断 |
| 代码耦合 | ✅ 每个功能独立成一个任务 |
| 功耗难降 | ✅ 空闲任务自动让 CPU 进低功耗 |
| 难扩展 | ✅ 加一个 `xTaskCreate` 就行 |

好，正式开始。

### 3.1 任务 —— 最基本的执行单元

**裸机**的时候：一个大 `while(1)` 里塞了一堆东西。

**RTOS 的思路**：加一层!

```c
// 裸机
while(1) {
    scan_key();
    update_led();
    uart_send();
}

// RTOS —— 加一层, 把一开始的循环整体, 拆成三个独立的任务
void task_key(void *pv) {
    while(1) { scan_key(); vTaskDelay(10); }
}

void task_led(void *pv) {
    while(1) { update_led(); vTaskDelay(50); }
}

void task_uart(void *pv) {
    while(1) { uart_send(); vTaskDelay(100); }
}
```

每个任务有自己的：
- **栈（Stack）** —— 存局部变量、函数调用
- **任务控制块（TCB）** —— 存优先级、状态、栈指针等信息
- **独立的逻辑** —— 互不干扰

那它如何代替**超级循环**呢?
接下来的分析中, 大家带着问题, 自会明晰

### 3.2 调度 —— 谁来跑、跑多久

任务这么多，CPU 只有一个，**听谁的？**

FreeRTOS 有三种调度方式：

| 调度方式 | 什么意思 | 类比 |
|----------|----------|------|
| **抢占式（Preemptive）** | 高优先级的任务就绪了，低优先级的立刻让 | 🏥 急诊插队 |
| **时间片轮转（Time Slicing）** | 同优先级轮流跑，每人分固定时长 | 🍚 食堂排队每人打一勺 |
| **合作式（Cooperative）** | 你不主动让，我就不抢 | 🧘 自觉让座 |

**FreeRTOS 默认就是抢占 + 时间片轮转**，大部分场景够用了。

#### 再说说优先级

上面一直在说"高优先级""低优先级"，那优先级**到底是什么**？

**优先级就是一个数字。** FreeRTOS 里数字越大，优先级越高：
> 但嵌入式中断模块里, 数字越小, 优先级越高

| 概念 | 说明 |
|------|------|
| 范围 | `0` ~ `configMAX_PRIORITIES - 1`（可配置，默认一般 5 或 32） |
| 0 | 最低——空闲任务（Idle Task）用的优先级 |
| 越大 | 越高，越容易被调度器选中 |
| 相同优先级 | 时间片轮转，轮流跑 |

```c
// 创建任务时通过 uxPriority 参数设置
xTaskCreate(task_led, "LED", 128, NULL, 2, NULL);   // 优先级 = 2
xTaskCreate(task_key, "KEY", 128, NULL, 1, NULL);   // 优先级 = 1
xTaskCreate(task_idle, "IDLE", 128, NULL, 0, NULL); // 优先级 = 0（空闲任务）
```

> 🍚 **记法：** 优先级 = 插队权。数字越大，越能插队。

那 `configMAX_PRIORITIES` 设多大合适？**别贪多。** FreeRTOS 内部维护一个**就绪列表数组**，大小就是 `configMAX_PRIORITIES`——不管你这个优先级有没有任务在用，**每个级别都分配了一个列表头**。设 32 级比设 5 级的 RAM 开销大好几倍：

```c
// FreeRTOS 源码里就是这么定义的（简化版）
static List_t pxReadyTasksLists[ configMAX_PRIORITIES ];
// ← 不管用没用，每个优先级都有一个 List_t
```

实际项目里 **5~8 个级别完全够用**，别设个 32 结果 RAM 炸了还不知道为什么()

### 3.3 状态切换 —— 任务都在干嘛

一个任务在任意时刻，一定处于以下四种状态之一：

```
┌───────────┐    运行态:
│  Running  │  ← 正在跑（占用 CPU）
|           |
├───────────┤    就绪态:
│   Ready   │  ← 准备好了，等 CPU 来宠幸
├───────────┤
|           |    阻塞态:
│  Blocked  │  ← 在等某件事（延时、等队列、等信号量）
|           |
├───────────┤    挂起态:
│ Suspended │  ← 被挂起，不参与调度
└───────────┘
```

状态之间怎么转换的？

```text
Ready     ──调度器选中──▶  Running
Running   ──时间片到／被抢──▶  Ready
Running   ──vTaskDelay／等资源──▶  Blocked
Blocked   ──延时到／资源到──▶  Ready
Running   ──vTaskSuspend──▶  Suspended
Suspended ──vTaskResume──▶  Ready
```
可在[状态切换](../excalidraws/状态切换.excalidraw)中, 仔细解释

### 3.4 ⭐ Blocked 状态 —— FreeRTOS 省 CPU 的关键


#### Q: 裸机的 delay 是什么？

#### A:

```c
while(1) {
    delay_ms(100);      // ← CPU 这 100ms 全在空转数数
    toggle_led();
}
```

单片机执行 `delay_ms(100)`，本质是 CPU 在一个循环里不断自增一个变量，直到计数值到。**这 100ms 里 CPU 不能响应任何其他任务**，因为你根本没写别的任务让它干。

#### Q: FreeRTOS 的 delay 又是什么？

#### A:

```c
void task_led(void *pvParameters) {
    while(1) {
        toggle_led();
        vTaskDelay(100);    // ← 任务说：我 Blocked 了，CPU 你拿去干别的
    }
}
```

`vTaskDelay(100)` 的执行流程：

```
① 调度器把这个任务从 Ready 队列 → 扔到 Blocked 队列
② 调度器立刻挑一个 Ready 的任务来跑
③ 那 100ms 里，CPU 在扫键盘、算数据、读传感器 —— 反正不是空转
④ 100ms 到了，这个任务自动回到 Ready 队列，等调度器下次选它
```

> 💡 **Blocked 状态的价值在于：当任务无话可做时，它主动交出 CPU，而不是让 CPU 闲着等它。省时间的不是『切换动作本身』，而是把『空等』变成了『干正事』。**

#### Q: 那「任务切换快」又省的是啥？

#### A:

省的是**切换本身的 overhead**：

```text
保存当前寄存器、栈指针
    → 找到下一个任务
    → 恢复它的寄存器、栈指针
```

FreeRTOS 在 ARM Cortex-M 上用 **PendSV 中断**实现，只需要几个汇编指令，**微秒级**完成。所以你不用担心切来切去浪费时间。

🤔 关于 PendSV 的原理，说实话我自己也是半懂不懂，后面启动流程那节会再聊到，这里先有个印象就行。

### 3.5 优先级反转 —— 一个经典坑

多任务调度看起来很美对吧？但它会引出一些反直觉的问题。

#### 场景

三个任务 + 一个共享资源：

| 任务 | 优先级 | 行为 |
|------|--------|------|
| 任务 H（High） | 🔴 最高 | 需要访问共享资源 |
| 任务 M（Medium） | 🟡 中等 | 纯计算，不碰共享资源 |
| 任务 L（Low） | 🟢 最低 | 先拿到了共享资源的锁 |

> 🤔 等等，这里说的"锁"是什么？
> 
> 在 RTOS 里，多个任务可能同时访问同一个共享资源（比如一个全局变量、一个外设寄存器）。如果任务 A 正在读数据读到一半，任务 B 把它改了——那 A 拿到手的数据就变成一半旧一半新，程序直接裂开 😵
> chovy,你们拿数据给我拿好的啊!
> 
> **锁（Mutex）就是用来解决这个问题的。** 任务在访问共享资源前先"上锁"（`xMutexTake`），用完了再"解锁"（`xMutexGive`）。其他任务如果也想用同一个资源，发现锁被别人拿着，资源被锁上, 就乖乖去 Blocked 排队等着，直到锁被释放。
> 
> 听着挺合理的对吧？但锁在多任务调度中也会引出一些反直觉的问题。下面就是一个经典踩坑流程——

#### 踩坑流程

```text
① 任务 L 拿到锁，进入共享资源区
② 任务 H 就绪了 —— 优先级高，直接抢占 L，H 开始跑
③ H 跑了一会儿，想用共享资源，
   发现锁被 L 拿着 → H 被 Blocked
④ 任务 M 就绪了！M 优先级比 L 高 → M 抢占 L
⑤ M 跑得欢，L 一直没机会释放锁 → H 就一直等不到锁
⑥ 结果：最高优先级的 H，被中优先级的 M 间接踩在脚下 😵
L、M：这就是我们的连携技！！！()
```

> **最高优先级的任务，反而在等最低优先级的任务释放锁。** 这就是**优先级反转**。

#### 💡 FreeRTOS 怎么解决：优先级继承

FreeRTOS 的互斥量（Mutex）自带一个机制叫 **优先级继承**：

```
① H 因为等 L 的锁而被 Blocked
② FreeRTOS 暂时把 L 的优先级提升到 H 的级别
③ L 获得 CPU → 快速运行 → 用完资源 → 释放锁
④ L 释放锁后 → 优先级降回原来
⑤ H 拿到锁 → 继续跑
```

> 🏃 **一句话：** 「优先级反转 ≈ 高速上快车道被慢车堵了，救护车在后面干着急，还被旁边车道的车加塞😡。优先级继承 ≈ 交警给慢车开道：『你快开走，别堵着！』——慢车麻溜办完让路，救护车通过✅」

<span style="color:red">注：</span> 用 `xSemaphoreCreateMutex()` 而不是 `xSemaphoreCreateBinary()` 创建互斥量，才自动带优先级继承。

| 函数 | 类型 | 优先级继承 | 适用场景 |
|------|------|-----------|---------|
| `xSemaphoreCreateMutex()` | 互斥量（Mutex） | ✅ 自带 | 保护共享资源（防止优先级反转） |
| `xSemaphoreCreateBinary()` | 二值信号量（Binary） | ❌ 没有 | 任务间的通知/同步（如"数据准备好了"） |

👉 **经验法则：** 保护共享资源（变量、外设）用 **Mutex**；通知事件（数据到了、传输完了）用 **Binary Semaphore**。~~混着用一时爽，debug 火葬场😇~~

### 3.6 任务间的同步与通信

多个任务之间怎么配合？FreeRTOS 提供了几种机制：

| 机制 | 干什么的 | 类比 |
|------|----------|------|
| **队列（Queue）** | 任务 A 发数据给任务 B | 羽毛球筒 —— A 往里扔，B 从里取 |
| **信号量（Semaphore）** | 通知某件事发生了（资源可用） | 🚦 红绿灯 —— 告诉你"可以走了" |
| **互斥量（Mutex）** | 同一时间只有一个任务能用某个资源 | 🔑 厕所钥匙 —— 谁拿谁进，别人等着 |
| **事件组（Event Group）** | 多个条件都满足才触发 | ✅✅ 打卡 —— 两个人都到了才出发 |

核心概念就一句话：**谁等谁、谁通知谁。**

#### 队列能干什么？

队列的核心本质其实就是**一个带阻塞机制和计数机制的 FIFO（先进先出）缓冲区**，所以还有很多玩法：

| 能力 | 怎么用 | 说明 |
|------|--------|------|
| **超时等待** | `xQueueReceive(..., portMAX_DELAY)` | 没数据时任务自动 Blocked，省 CPU |
| **覆盖写入** | `xQueueOverwrite()` | 只保留最新值，丢掉旧数据（适合传感器读数） |
| **只看不取** | `xQueuePeek()` | 偷看队首数据，不移除 |
| **中断中发收** | `xQueueSendFromISR()` | ISR 里也能安全地用队列(普通队列无法在中断中使用) |
| **多对多** | 多发多收 | 多个任务往同个队列发、多个任务从同个队列收 |
| **队列集** | `xQueueCreateSet()` | 同时等多个队列/信号量，谁先来先处理 |

> **一句话：** 队列是 FreeRTOS 里最通用的通信原语.


#### 一个有意思的细节：信号量 / 互斥量 的底层是队列

你可能会想，队列和信号量是两种东西吧？其实 **FreeRTOS 的信号量底层就是队列** —— 只不过是一种特殊的队列。

```c
// 普通队列 —— 每个消息携带实际数据
xQueueCreate(5, sizeof(int));  // 能存 5 个 int

// 信号量 —— 「队列项大小」为 0
xQueueCreate(1, 0);  // 二进制信号量：能 "存" 1 个，但不存任何实际数据
xQueueCreate(5, 0);  // 计数信号量：能 "存" 5 个，同样不存数据
```

把 `uxItemSize` 设为 `0`，队列就变成了一个**只计数、不存数据**的东西：

- **给信号量**（`xSemaphoreGive`）→ 计数 +1，不拷贝任何数据
- **取信号量**（`xSemaphoreTake`）→ 计数 −1，不读取任何数据

> 💡 **好处：** 省 RAM（不需要数据缓冲区）+ 省 CPU（不需要数据拷贝）。轻量又高效，很适合单片机。


 
### 3.7 总结——FreeRTOS 的设计哲学

通篇讲下来，FreeRTOS 的设计思想其实可以总结成几条：

1. **加一层 —— 任务抽象**
   把裸机一个大 `while(1)` 拆成多个独立任务。每个任务只管自己的事，**解耦** —— 改按键逻辑不用碰 LED 代码，加新功能就加个新任务。这是最根本的思想转变。

2. **让出 CPU —— Blocked 机制**
   任务等资源的时候不空转，主动阻塞交出 CPU。空闲任务再让芯片睡大觉 —— 省电又省 CPU。这是 RTOS 比裸机 delay 高明的地方。

3. **谁急谁上 —— 抢占式调度**
   高优先级任务就绪了，低优先级立马让。紧急事件不排队，保证实时性。但小心优先级反转 💀
4. **统一通信原语 —— 队列**
   队列不仅能传数据，还能当信号量、互斥量用。一套机制走天下，简洁而强大

5. **内核最小化**：只做调度和同步，不绑协议栈、文件系统、设备驱动。

6. **可裁剪**：`FreeRTOSConfig.h` 里一堆 `#define`，用哪个开哪个，不用就注释掉，省 ROM。

7. **面向嵌入式**：支持静态内存分配，你不用非得用 `malloc`（~~单片机没有 MMU，malloc 用起来有种裸奔的美~~）

## 4. FreeRTOS vs 多线程编程

上文提过, 我想分享FreeRTOS的很大原因也是因为它和现代操作系统的关联比较大——尤其对学过后端、操作系统的同学来说, FreeRTOS 的任务机制本质上就是**一个轻量级的多线程系统**, 很多概念是相通的。
(如果是上学期讲这一节例会, 我觉得效果会更好些, 部分同学现在已经了解完这些概念了, 🤔权当是巩固罢😶‍🌫️)

### FreeRTOS 的 task 就是线程

如果你学过 POSIX 线程（pthread）或 Java 的 Thread，你会发现 FreeRTOS 的任务和它们惊人地相似：

```c
// pthread — 创建一个线程
pthread_create(&tid, NULL, thread_func, arg);

// FreeRTOS — 创建一个任务
xTaskCreate(task_func, "Task", 128, arg, 1, &handle);
```

| 概念 | FreeRTOS | pthread / 多线程 |
|------|----------|-----------------|
| **执行单元** | Task（任务） | Thread（线程） |
| **创建** | `xTaskCreate()` | `pthread_create()` |
| **结束** | 函数 return 或 `vTaskDelete()` | 函数 return 或 `pthread_exit()` |
| **栈** | 每个任务独立栈 | 每个线程独立栈 |
| **共享地址空间** | ✅ 所有任务共享同一内存 | ✅ 同一进程内线程共享内存 |
| **同步** | 互斥量、信号量、队列 | 互斥锁、信号量、条件变量 |
| **调度** | 抢占式 + 时间片轮转 | 抢占式 + 时间片（OS 决定） |

> 所以说，**在 FreeRTOS 里学到的多任务思想，直接可以平移到你以后学多线程编程的场景上。**

### 但差异也不小

虽然思路一样，但 FreeRTOS 和 PC 上的多线程有几个关键区别：

| 维度 | FreeRTOS | PC 多线程（Linux/Windows） |
|------|----------|--------------------------|
| **MMU / 地址空间** | ❌ 没有 MMU，所有任务裸奔在同一个地址空间 | ✅ 进程隔离 + 虚拟地址，一个线程崩了不一定影响整个进程 |
| **用户态 / 内核态** | ❌ 不区分，任务和内核跑在同一个权限级 | ✅ 系统调用切换用户态/内核态，安全性高 |
| **调度目标** | 🎯 **确定性** —— 高优先级任务必须及时响应 | 🎯 **公平性** —— 所有线程尽量分到等量的 CPU |
| **内存分配** | 谨慎用 `malloc`，容易碎片化，提供静态分配方案 | `malloc` 是线程安全的（TLSF/ptmalloc），相对放心 |
| **栈空间** | 任务栈在编译时就定死了，溢出就死机 | 线程栈有保护页（Guard Page），溢出能检测 |
| **标准库** | 没有完整的 libc 支持，很多标准库函数不可用 | 完整的 libc / stdlib |

### 值得讲的点

- **学 FreeRTOS 的最大价值**不是学会怎么调 API（那玩意儿查手册就行），而是理解 **"多任务是怎么协作的"**——这个能力放之四海而皆准。
- 反过来学：如果你已经学过操作系统的多线程，你会发现 FreeRTOS 的这些概念**异常熟悉**。它就是操作系统的一个**极简实现**，反而因为没有 MMU、没有用户态这些"噪声"，**让你更看清多任务调度的本质**。

## 5. 启动与文件结构

### 5.1 完整执行流

单片机从上电到跑起 FreeRTOS，历经这么几步：

```text
① 上电 → 芯片从 Flash 取出 Reset_Handler 地址
② Reset_Handler 执行：
      - 复制 .data 段（初始化的全局变量）到 RAM
      - 清零 .bss 段（未初始化的全局变量）
      - 设置堆栈指针 SP
③ 跳转到 SystemInit（配置系统时钟、PLL 锁相环）
④ 跳转到 main()
```

这些是编译器 + 芯片启动文件帮你做的，知道一下就行。

### 5.2 main() 里面发生了什么

这是最值得展开讲的一段：

```text
main()
  │
  ├── 外设初始化（GPIO、UART、定时器等）
  │
  ├── 创建任务
  │     xTaskCreate(Task1, "Task1", 128, NULL, 1, NULL);
  │     xTaskCreate(Task2, "Task2", 128, NULL, 2, NULL);
  │     ↑ 注意！这只是在内存里建了个任务控制块（TCB）
  │       任务还没开始跑，只是注册了一下
  │
  ├── vTaskStartScheduler()   ← 这一行是分水岭！
  │     │
  │     ├── 创建空闲任务（Idle Task）—— CPU 没事干就待在这
  │     ├── 如果有软件定时器 → 创建定时器服务任务
  │     ├── 初始化系统时钟节拍（SysTick，一般是 1ms 中断一次）
  │     ├── 启动第一个任务（手动触发 PendSV 中断）
  │     └── 不再return!!!!!
  │
  └── 永远不会执行到这里
```

> 🔑 **一句话：** 「`vTaskStartScheduler()` 就像按下了一个开关，从此不再是你的 `main` 在控制 CPU，而是调度器在管。」

### 5.3 vTaskStartScheduler() 之后的世界
`vTaskStartScheduler()`函数内,先是初始化部分设备, 最后将CPU的指令读取地址指向第一个要执行的任务, 此后退下舞台, 之后调度职权, 由中断全权接管.


| 事件 | 谁触发 | 干啥 |
|------|--------|------|
| **SysTick 中断** | 硬件定时器（1ms 一次） | 更新任务延时计数 + 触发调度决策 |
| **PendSV 中断** | 调度器软件触发 | 保存旧任务上下文 → 加载新任务上下文 |
| **SVC 中断** | 任务调用 FreeRTOS API | 从用户模式切到内核模式执行系统调用 |

系统跑起来以后，就是一个**中断驱动**的世界：
- **SysTick** 像心跳，1ms 跳一次，告诉调度器"该看看有没有任务等完了"
- **PendSV** 像调度器的"换人按钮" —— 需要切换任务时就触发它
- **SVC** 像"系统调用" —— 任务想调用 FreeRTOS 的服务时就触发它
