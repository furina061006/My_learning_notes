## 二、背景：为什么需要 RTOS

> ⚠️ **注意：听众可能完全不懂裸机编程，这一节要讲得非常基础。**

### 2.1 先聊：单片机是怎么跑程序的？

**你写的 C 代码 → 编译器 → 变成二进制机器码 → 烧进 Flash → 上电后 CPU 一条一条取指令执行**

大致上就是：

```
main() {
    while(1) {
        step1();
        step2();
        step3();
    }
}
```

永远在这个圈里打转，这叫 **超级循环（Super Loop）**，也是**裸机编程**最典型的样子。

### 2.2 这样做有什么问题？

#### 问题一：CPU 在「等」的时候啥也没干

```
while(1) {
    按键扫描();     ← 几微秒
    delay_ms(10);  ← CPU 在这 10ms 里空转数数！！！
    LED刷新();
    串口处理();
}
```

函数调用的执行顺序完全是**串行**的。当 `delay_ms(10)` 在跑的时候，CPU 就是在数数：
- 一个大循环 `for(i=0; i<100000; i++);` 
- 这 10ms 里，**闹钟按键按了？CPU 不知道。传感器数据来了？CPU 不知道。**
- 因为 CPU 在忙着数数呢。

> **讲给听众：** 这就像一个厨师在等水烧开的时候盯着水壶发呆，而不是趁这个时间切菜、洗锅。

#### 问题二：实时性差 —— 紧急事件得排队等

假如有这样一个需求：
- 一个按键按下时，必须在 **5ms 内** 响应
- 但现在主循环里有一个任务跑得特别久（比如刷一个 OLED 屏幕，可能要 20ms）

```c
while(1) {
    oled_display();   ← 要跑 20ms
    scan_key();       ← 按键检测被排在后面！！！
    process_sensor(); 
}
```

用户按了键，但是 OLED 没刷完之前，CPU 根本不会去扫键盘。**响应时间由最长的那个任务决定。**

> **讲给听众：** 就像排队办事，前面那个人办了一个超复杂的业务（20分钟），你只是交个表（5秒），但你得等他办完。

#### 问题三：代码耦合 —— 加功能就要动 main 循环

```
while(1) {
    scan_key();
    update_led();
    uart_send();
    sensor_read();
    // 想加一个功能 → 在这里加一行
    // 想删一个功能 → 又动这里
}
```

- 所有功能粘在一起
- 一个模块崩了 → while 循环卡住 → 整个系统死掉
- 新人来看代码，一个 while 里几十个函数调用，想死

#### 问题四：功耗控制困难

想省电怎么办？在裸机里你可能会这样做：

```c
while(1) {
    do_something();
    delay_ms(100);
    // CPU 还在跑，功耗降不下来
    // 进入睡眠模式？醒来又要重新初始化…
}
```

芯片就算不需要干活，也必须一直在「跑」的状态，才能保证随时响应。电池设备的话就哭了。

### 2.3 总结：痛点列表

| 痛点 | 一句话 |
|------|--------|
| CPU 浪费 | delay 在空转，没干正事 |
| 实时性差 | 紧急事件得等前面的跑完 |
| 代码耦合 | 所有功能塞在一个 while 里 |
| 功耗难降 | CPU 不能闲下来 |
| 难扩展 | 加一个功能牵一发动全身 |

### 2.4 于是 RTOS 来解决这些问题

| 痛点 | RTOS 怎么解决 |
|------|---------------|
| CPU 浪费 | ✅ Blocked 状态主动交出 CPU（详见 3.4） |
| 实时性差 | ✅ 抢占式调度，高优先级任务随时打断 |
| 代码耦合 | ✅ 每个功能独立成一个任务，互不干扰 |
| 功耗难降 | ✅ 空闲任务自动让 CPU 进低功耗模式 |
| 难扩展 | ✅ 加一个任务就是加一个 xTaskCreate，不影响别的 |

## 三、FreeRTOS 是什么

### 一句话

> **FreeRTOS 是一个为单片机设计的、轻量级的、开源的实时操作系统内核。**

### 详细介绍

| 方面 | 内容 |
|------|------|
| **出身** | 2003年由 Richard Barry 开发，后来被 Amazon 收购，现在是 Amazon 旗下项目 |
| **名字含义** | **Free**（免费）+ **RTOS**（实时操作系统） |
| **语言** | 纯 C 实现，移植性好 |
| **体量** | 内核最小只需 **3~5 KB ROM**，RAM 几 KB 就能跑 |
| **芯片支持** | ARM Cortex-M、RISC-V、AVR、PIC、MSP430…… 几乎覆盖所有主流单片机 |
| **开源协议** | MIT，商用也没问题 |
| **生态** | 官方有 AWS IoT 集成，社区资源极其丰富，中文资料也多 |

### 为什么要选 FreeRTOS？

市面上还有很多别的 RTOS（uC/OS、RT-Thread、RTEMS……），但 FreeRTOS 的优势很明显：

1. **最轻量** —— ROM/RAM 占用是所有 RTOS 里最小的之一
2. **最普及** —— 资料最多，遇到问题百度/谷歌一下就有答案
3. **大厂背书** —— Amazon 维护，TrustZone、SMP 多核都在逐步支持
4. **完全免费** —— 不像 uC/OS 商用要收费

> **讲给听众：**
> 「FreeRTOS 在单片机 RTOS 里的地位，差不多就像 Linux 在服务器操作系统的地位。它不一定是最强最全的，但它是你**大概率会遇到的第一个**。」

- [ ] **听众可能对中断、堆栈概念模糊** —— 穿插在各模块中简要说明

## 四、核心设计思想（详细展开）

### 3.1 任务 —— 最基本的执行单元

把一个大 `while(1)` 拆成几个独立的小 `while(1)`，每个任务有自己的栈、自己的状态。

### 3.2 调度 —— 谁来跑、跑多久

三种调度方式：

| 调度方式 | 什么意思 | 类比 |
|----------|----------|------|
| **抢占式（Preemptive）** | 高优先级的任务就绪了，低优先级的立刻让 | 急诊插队 |
| **时间片轮转（Time Slicing）** | 同优先级轮流跑，每人分固定时长 | 食堂排队每人打一勺 |
| **合作式（Cooperative）** | 你不主动让，我就不抢 | 自觉让座 |

FreeRTOS 默认是抢占 + 时间片轮转。

#### ⚠️ 优先级反转（Priority Inversion）—— 一个经典坑

这就是优先级调度会引出的一个反直觉问题。

**场景（三个任务 + 一个共享资源）：**

| 任务 | 优先级 | 行为 |
|------|--------|------|
| 任务 H（High） | 🔴 高 | 需要访问共享资源 |
| 任务 M（Medium） | 🟡 中 | 纯计算，不碰共享资源 |
| 任务 L（Low） | 🟢 低 | 先拿到了共享资源的锁 |

**踩坑流程：**

```
① 任务 L 拿到锁，进入共享资源区
② 任务 H 就绪，抢占 L —— H 开始跑
③ H 想要锁，发现锁被 L 拿着 → H 被 Blocked
④ 此时任务 M 就绪啦！M 优先级比 L 高 → M 抢占 L
⑤ M 跑得欢，L 一直没机会释放锁 → H 等不到锁
⑥ **结果：最高优先级的 H，被中优先级的 M 间接踩在脚下**
```

最高优先级的任务反而在等最低优先级的任务 —— 这就是**优先级反转**。

#### 💡 FreeRTOS 怎么解决：优先级继承

当 H 因为等 L 的锁而被 Blocked 时：

1. FreeRTOS **暂时把 L 的优先级提升到 H 的级别**
2. L 获得 CPU 快速运行，尽快用完资源释放锁
3. L 释放锁后，优先级降回原来的低优先级
4. H 拿到锁继续跑

**去翻一下代码：** 用 `xSemaphoreCreateMutex()` 而不是 `xSemaphoreCreateBinary()` 创建互斥量，就自动带优先级继承。

> **一句话讲给听众：**
> 「优先级反转就是『高等公民被中等公民插队气死』。优先级继承就是让低等公民临时获得高等身份，快速办完事让位。」

### 3.3 状态切换 —— 任务都在干嘛

四种状态：

```
Running  →  正在跑（占用 CPU）
Ready    →  准备好了，等 CPU
Blocked  →  在等某件事（延时、等队列、等信号量）
Suspended → 被挂起，不参与调度
```

### 3.4 ⭐ Blocked 状态 —— FreeRTOS 省 CPU 的关键

这是最值得讲清楚的一个点，很多入门的人没意识到。

#### 裸机的 delay 是什么？

```
while(1) {
    delay_ms(100);      ← CPU 这100ms 全在空转数数
    toggle_led();
}
```

单片机执行 `delay_ms(100)` 本质是：**CPU 在一个循环里不断自增一个变量，直到计数值到，啥正事也没干**。这 100ms 里 CPU 不能响应任何其他任务。

#### FreeRTOS 的 delay 是什么？

```
void task_led(void *pv) {
    while(1) {
        toggle_led();
        vTaskDelay(100);    ← 任务说：我现在 Blocked 了，CPU 你拿去干别的
    }
}
```

`vTaskDelay(100)` 的执行流程：
1. 把这个任务从 Ready 队列移到 **Blocked 队列**
2. 调度器立刻挑一个 **Ready 状态的任务** 来跑
3. 那 100ms 里 CPU 在干正事（扫键盘、算数据、读传感器），不是空转
4. 100ms 到了，这个任务自动移回 Ready 队列，等调度器下次选它

#### 一句话总结

> **Blocked 状态的价值在于：当任务无话可做时，它主动交出 CPU，而不是让 CPU 闲着等它。省时间的不是『切换动作本身』，而是把『空等』变成了『干正事』。**

#### 那「任务切换快」省的是什么？

切换快（上下文切换快）省的是 **切换本身的 overhead**：

```
保存当前寄存器、栈指针
    → 找到下一个任务
    → 恢复它的寄存器、栈指针
``` 

FreeRTOS 在 ARM Cortex-M 上用 PendSV 中断实现，只需几个指令，微秒级完成。

---

### 3.5 任务间的同步与通信

- **队列（Queue）**：任务 A 发数据给任务 B
- **信号量（Semaphore）**：通知某件事发生了（资源可用）
- **互斥量（Mutex）**：同一时间只有一个任务能用某个资源（防止打架）
- **关键概念**：谁等谁、谁通知谁

### 3.6 FreeRTOS 的设计哲学

- **内核最小化**：只做调度和同步，不绑协议栈、文件系统、设备驱动
- **可裁剪**：`FreeRTOSConfig.h` 里 `#define` 开关，用哪部分就编译哪部分
- **面向嵌入式**：静态内存分配选项，不用 malloc（适合没有 MMU 的单片机）

## 五、FreeRTOS vs 现代操作系统

**现代 OS 能干的：**
- 多任务 / 进程管理
- 文件系统
- 网络协议栈
- 虚拟内存 / MMU
- 设备驱动框架
- GUI 系统

**FreeRTOS 实现了其中的哪些（简化版）：**

| 能力 | RTOS | 现代 OS |
|------|------|---------|
| 多任务 | ✅ 抢占式 + 时间片 | ✅ 优先级 + 时间片 |
| 同步/通信 | ✅ 队列、信号量、互斥量 | ✅ 管道、信号量、共享内存 |
| 内存管理 | ✅ 简单的 heap 方案（5种） | ✅ 虚拟内存 / 分页 |
| 上下文切换 | ✅ 寄存器级 | ✅ 进程上下文 + TLB 刷新 |
| 中断管理 | ✅ 延迟中断处理（ISR + 任务配合） | ✅ 内核态中断处理 |
| 文件系统 | ❌ 不内置 | ✅ 标配 |
| 网络协议 | ❌ 不内置（可选 Plus/TCP） | ✅ 标配 |
| 虚拟内存 | ❌ 没有 MMU，用不了 | ✅ 核心能力 |
| 用户/内核态隔离 | ❌ 内核和任务在同一个地址空间 | ✅ 核心安全机制 |
| GUI | ❌ 不提供 | ✅ 各系统不同 |

**可讲的点：**
- FreeRTOS 就是一个「精简到只剩多任务核心」的微型 OS 内核
- 它没有用户态/内核态隔离，所有任务跑在同一个地址空间（因为单片机本身就是跑在特权模式）
- 没有 MMU → 没有虚拟内存 → 也没有进程隔离 → 一个任务写越界能搞崩整个系统
- 但也因此足够轻量，RAM 几 KB 就能跑

## 六、启动与文件结构

### 完整执行流

```
① 上电 → 芯片从 Flash 取出 Reset_Handler 地址
② Reset_Handler 执行：
      - 复制 .data 段（初始化的全局变量）到 RAM
      - 清零 .bss 段（未初始化的全局变量）
      - 设置堆栈指针 SP
③ 跳转到 SystemInit（配置系统时钟、PLL锁相环）
④ 跳转到 main()
```

### main() 里面发生了什么

这是最值得展开讲的一段流程：

```
main()
  │
  ├── 外设初始化（GPIO、UART、定时器等）
  │
  ├── 创建任务
  │     xTaskCreate(Task1, "Task1", 128, NULL, 1, NULL);
  │     xTaskCreate(Task2, "Task2", 128, NULL, 2, NULL);
  │     ↑ 这只是在内存里建了个任务控制块（TCB），还没开始跑
  │
  ├── vTaskStartScheduler()   ← 这一行是分水岭
  │     │
  │     ├── 创建空闲任务（Idle Task）
  │     ├── 如果有软件定时器 → 创建定时器服务任务
  │     ├── 初始化系统时钟节拍（SysTick，一般是 1ms 中断一次）
  │     ├── 启动第一个任务（手动触发 PendSV 中断）
  │     └── 不再返回！（从此控制权交给调度器）
  │
  └── 永远不会执行到这里
```

### vTaskStartScheduler() 之后的世界

| 事件 | 谁触发 | 干啥 |
|------|--------|------|
| **SysTick 中断** | 硬件定时器（1ms 一次） | 更新任务延时计数 + 触发调度决策 |
| **PendSV 中断** | 调度器软件触发 | 保存旧任务上下文 → 加载新任务上下文 |
| **SVC 中断** | 任务调用 FreeRTOS API | 从用户模式切到内核模式执行系统调用 |

> **一句话讲给听众：**
> 「`vTaskStartScheduler()` 就像按下了一个开关，从此不再是你的 main 在控制 CPU，而是调度器在管。就像交了班，你自己站到旁边去了。」

### 问题：为什么 FreeRTOS 非要用中断来做任务切换？

而不是直接在代码里写一个函数调用来切换？

```
如果用普通函数调用：
    vTaskSwitch() → 切换到下一个任务...
    → 问题：函数调用会压栈，切换完栈指针都不对了，
       return 回到哪去了？没法搞。

用 PendSV 中断：
    → 中断有硬件压栈（自动保存 R0-R3, R12, LR, PC, xPSR）
    → PendSV 优先级设最低，不会和真正的中断抢
    → 中断返回时自动恢复新任务的上下文
    → 完美衔接
```

**所以 PendSV 是 ARM Cortex-M 上 RTOS 的基础设施**，不是 FreeRTOS 独有的，别的 RTOS 也这么干。
